## Data strategy

**Recommended source: Zerodha Kite + `yfinance` as fallback.** Even for a free start, open a Zerodha demat account — the Kite Connect API (₹2,000/month) gives you clean 1-minute OHLCV with corporate-action-adjusted data going back to ~2007, and critically, the same API doubles as your live feed later. ## Feature engineering (what actually works)

The rule-based community uses a well-validated core set. Build these into a `FeatureBuilder` class:

**Price-based:** EMA(9), EMA(21), EMA(200) for trend regime; RSI(14); MACD(12,26,9); Bollinger Bands(20,2); ATR(14) for volatility sizing; Supertrend(10,3) — very popular in Indian retail algo trading.

**Volume-based:** VWAP (intraday anchor); Relative volume (today's vol vs 20-day avg at same time); On-Balance Volume; delivery percentage from NSE (surprisingly predictive for swing).

**Multi-timeframe stacking:** For each 1-minute candle, compute features at 1m, 5m, and 15m and concatenate as a single feature vector. Add boolean flags like `ema9_above_ema21_on_15m` — these act as regime filters and dramatically reduce false signals.

**Target label construction:** Use a forward-looking window. For a candle at time `t`, label = `BUY` if price rises >0.5% within next 15 candles without hitting a 0.3% stop, else `SELL`/`HOLD`. This is the standard approach used by most published NSE ML papers.

## The three models — shared pipeline

All three models consume the **same feature matrix** and output a 3-class probability vector `[P_buy, P_sell, P_hold]`.

**LSTM:** Input shape `(lookback=60, n_features)`. Two stacked LSTM layers (128, 64 units), dropout 0.3, dense head with softmax. Good at capturing sequential dependencies — opening range breakout patterns, gap-up continuation.

**GRU:** Same input shape, but single GRU layer (128 units) + dense. Trains faster, often matches LSTM on short intraday timeframes. Use this as your "quick signal" model.

**LightGBM:** Flattened feature vector (no sequence — just the current candle's full feature set). Add lagged features manually (t-1, t-5, t-10). Excellent at regime classification — whether the market is trending, ranging, or volatile. This model acts as a **meta-filter** for the neural nets.

**Shared pipeline components your coding model needs:**

```
DataLoader → FeatureBuilder → Normalizer (per-ticker rolling z-score, 
not global — avoids lookahead) → SequenceBuilder → Train/Val/Test split 
(time-based, never random shuffle — that's data leakage)
```

## Backtesting framework

Use **`vectorbt`** (not `backtrader` — it's vectorized, handles multi-ticker, and produces beautiful HTML reports). For walk-forward optimization, implement a custom `WalkForwardOptimizer` class:

- Split the 10–20 years into rolling windows: 18 months train, 6 months test, step by 3 months
- Retrain each model on each training window, evaluate on the out-of-sample test window
- This is how professional quants validate — not a single backtest

**Key metrics to report per model:** Sharpe ratio, Calmar ratio, max drawdown, win rate, avg profit/loss ratio, number of trades per day. A Sharpe >1.5 and max drawdown <15% is a realistic target for intraday NSE.

**India-specific backtest realism checklist:**
- Apply 0.03% brokerage + 0.1% STT (Securities Transaction Tax) on sell side + SEBI charges — ignoring these inflates returns by ~0.2% per trade which destroys profitability at high frequency
- Apply realistic slippage: 0.05% per trade for Nifty 50 stocks, 0.1–0.15% for Nifty 500 mid-caps
- Enforce NSE trading hours: 9:15 AM – 3:30 PM IST; no trades in first 5 minutes (opening auction noise)
- Mark all market holidays from NSE calendar

## Signal fusion (the strategy layer)

Don't just take the LSTM signal at face value. Stack the three model outputs through a **rule-based filter**:

1. **LightGBM says "trending regime"** → allow LSTM/GRU BUY signals through
2. **LightGBM says "ranging regime"** → suppress breakout signals, allow mean-reversion signals
3. **Signal must have consensus ≥ 2/3 models** with confidence > 0.65 to become actionable
4. **Volume confirmation:** current volume must be > 1.5× its 20-period average at signal time

This ensemble + filter approach is what separates backtested strategies that work in live markets from those that don't.

## Crash-proofing the training notebook

This is the most important part. Tell your coding model to implement these patterns:

**Epoch checkpointing with resume logic:**
```python
# At start of training, check if a checkpoint exists
import os
start_epoch = 0
ckpt_path = MODEL_DIR + 'lstm_latest.pt'
if os.path.exists(ckpt_path):
    checkpoint = torch.load(ckpt_path)
    model.load_state_dict(checkpoint['model_state'])
    optimizer.load_state_dict(checkpoint['optimizer_state'])
    start_epoch = checkpoint['epoch'] + 1
    print(f"Resumed from epoch {start_epoch}")

# At end of each epoch, save
torch.save({
    'epoch': epoch,
    'model_state': model.state_dict(),
    'optimizer_state': optimizer.state_dict(),
    'val_loss': val_loss,
}, ckpt_path)
```

## GPU-specific training tips for Colab

Tell your coding model to always:

- Use `device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')` and move model + tensors to device
- Set `pin_memory=True, num_workers=2` in DataLoader for faster GPU feeding
- Use `torch.cuda.empty_cache()` between training the three models — Colab's T4 has 15GB but your feature matrices can be large
- Use **mixed precision** (`torch.cuda.amp.autocast()`) — cuts memory by ~40% and speeds training by 30–50% with no accuracy loss on this type of task

For LightGBM, pass `device='gpu'` in the params dict — Colab's CUDA supports it natively.

The `feature_config.json` is critical — it stores the exact list and order of features the model was trained on, so the live inference pipeline builds features in exactly the same way. This is the most common source of silent bugs in production.
